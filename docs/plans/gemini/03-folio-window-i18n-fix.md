# План исправления локализации Folio Windows (i18n)

## Проблема
На скриншоте видно, что названия окон фолио (Folio Windows) отображаются на русском языке («Основной», «Компания»), несмотря на то, что включена английская локализация интерфейса. 

Причина кроется в том, что эти строки жестко зашиты (hardcoded) на уровне базы данных и API:
1. В схеме `packages/db/src/schema/financial.ts` значение по умолчанию — `"Основной"`.
2. В API `apps/api/src/routes/bookings.ts` при создании бронирования вставляется окно с лейблом `"Основной"`.
3. В сидировании БД `packages/db/src/seed.ts` окна создаются как `"Основной"` и `"Компания"`.

Поскольку фронтенд просто отрисовывает то, что пришло из базы данных (`window.label`), переключение локали на UI не влияет на эти строки.

---

## План решения

### Шаг 1. Обновление словарей (Dictionaries)
Добавить новые ключи для системных названий окон в i18n словари.
- **`en.ts`**: 
  ```typescript
  "folio.windowMain": "Main",
  "folio.windowCompany": "Company",
  ```
- **`ru.ts`**:
  ```typescript
  "folio.windowMain": "Основной",
  "folio.windowCompany": "Компания",
  ```

### Шаг 2. Защита на фронтенде (UI Translation)
В компоненте `apps/web/src/app/bookings/[id]/folio-section.tsx` добавить функцию-хелпер для локализации известных системных лейблов. 

```typescript
function translateWindowLabel(label: string, dict: Dictionary): string {
  // Обрабатываем как новые английские (Main/Company), так и старые русские записи для обратной совместимости
  if (label === "Main" || label === "Основной") return t(dict, "folio.windowMain");
  if (label === "Company" || label === "Компания") return t(dict, "folio.windowCompany");
  
  // Если это кастомное название, созданное пользователем, возвращаем как есть
  return label; 
}
```
И использовать эту функцию при рендере вкладок `W1: {translateWindowLabel(w.label, dict)}`.

### Шаг 3. Исправление источника правды (DB & API)
Перевести систему на хранение каноничных английских значений в базе данных по умолчанию.
1. **`packages/db/src/schema/financial.ts`**: изменить `default("Основной")` на `default("Main")`. Запустить генерацию миграции.
2. **`apps/api/src/routes/bookings.ts`**: изменить хардкод `"Основной"` на `"Main"`.
3. **`packages/db/src/seed.ts`**: изменить данные для сидирования на `"Main"` и `"Company"`.

---

## Защита логики фикса (Defense of the Approach)

1. **Почему мы не переводим на бэкенде?**
   Бэкенд должен оставаться независимым от локали пользователя (locale-agnostic). Локаль — это настройка отображения (presentation layer), за которую отвечает фронтенд. Бэкенд возвращает каноничные данные.

2. **Почему в базе храним английские слова "Main", а не ключи перевода типа "folio.main"?**
   В полноценных PMS названия окон фолио (Folio Windows) часто могут переименовываться пользователями (например, "Расходы на бар", "Счет фирмы ООО Ромашка"). То есть поле `label` является гибридным: оно содержит либо системный дефолт, либо пользовательский ввод. Если мы начнем хранить ключи перевода в БД, нам понадобится дополнительное поле `is_system_label`, чтобы отличать ключи от пользовательского ввода. 
   Текущий подход (перехват известных слов `Main` и `Company` на фронтенде) решает задачу локализации системных значений без усложнения схемы базы данных и позволяет корректно выводить пользовательские названия "как есть".

3. **Зачем обрабатывать старые русские значения "Основной" на фронте?**
   Для обеспечения обратной совместимости (Backwards Compatibility). В базе данных уже могут существовать записи, созданные до этого фикса. Если фронтенд будет искать только `"Main"`, старые бронирования так и останутся на русском у англоязычных пользователей. Обработка обоих вариантов гарантирует, что весь исторический системный мусор будет красиво локализован.
