# Промт для GLM5.1: Исследование — логика расчёта Room Tax в ночном аудите

## Контекст
Проект: PMS на Fastify 5 + TypeScript + Drizzle ORM + PostgreSQL.
Монорепо: `apps/api/src/routes/night-audit.ts`, `packages/db/src/schema/`.

## Наблюдение
После ночного аудита для бронирования GBH-000004 (Corporate Rate, 4500₽/ночь):
```
2026-04-11  ROOM       Room Charge(auto)    4 500,01
2026-04-11  ROOM_TAX   Room Tax(auto)         900,00
2026-04-11  BREAKFAST  Breakfast for Company  500,00
```

ROOM_TAX = 900₽ при ROOM = 4500₽ → ровно 20%.

## Вопросы для исследования
1. Где в коде определяется ставка Room Tax? Найди в `apps/api/src/routes/night-audit.ts` или в `packages/domain/` логику расчёта налога.
2. Как ставка налога конфигурируется? Это хардкод (20%) или берётся из БД (таблица transaction codes / rate plans / property settings)?
3. В Opera PMS налог обычно настраивается через Transaction Codes с типом `tax` и процентом от base charge. Проверь, есть ли такая логика в нашей схеме.
4. Правильно ли, что налог начисляется на полную стоимость номера включая пакет (BREAKFAST)? Или только на Room Charge?
5. Сумма 4500,01 выглядит странно (почему не ровно 4500,00) — найди источник дробной копейки.
