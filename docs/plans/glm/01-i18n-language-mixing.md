# Промт для GLM5.1: Исследование — смешение языков в UI

## Контекст
Проект: открытая PMS (Property Management System) — замена Oracle Opera PMS V5.
Стек: Next.js 16 (App Router) + React 19 + Tailwind 4, API на Fastify 5 + TypeScript.
Монорепо: `apps/web/` (фронтенд), `apps/api/` (бэкенд).

## Проблема
В UI одновременно присутствуют русские и английские строки. Примеры:
- Кнопки: "Check In", "Check Out" — английские
- Статусы: "Checked In", "Due In", "Due Out" — английские
- Предупреждения и сообщения — русские
- Лейблы полей — смесь
- Диалог "Check-in в грязный номер" — частично на русском, частично на английском

Это создаёт хаос в интерфейсе и мешает демонстрации продукта.

## Задача для исследования
1. Пройдись по `apps/web/src/` и найди все места, где есть хардкоженные строки (на русском и английском).
2. Найди все компоненты в `apps/web/src/app/bookings/` и `apps/web/src/components/` — зафиксируй какие строки в каком языке.
3. Изучи, есть ли уже какая-то i18n-библиотека в `apps/web/package.json` (next-intl, i18next, react-i18next и т.п.).
4. Предложи стратегию локализации с учётом реалий:
   - Целевой язык интерфейса: **русский**
   - Технические термины отрасли (Check In, Check Out, Folio, Due In, Due Out, Night Audit) — оставить на английском как устоявшиеся термины
   - Все системные сообщения, лейблы форм, описания — перевести на русский
5. Дай конкретный план миграции: что менять, в каком порядке, какую библиотеку использовать (или обойтись без библиотеки).

## Ожидаемый результат
Markdown-документ с:
- Инвентаризацией строк по компонентам (таблица: файл, строка, язык, рекомендация)
- Выбором подхода (с библиотекой / без)
- Пошаговым планом реализации
- Примером для одного компонента (до/после)
